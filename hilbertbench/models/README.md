# `hilbertbench/models/` — Generated Data Models

**Do not hand-edit the Python files in this directory.** They are
generated from JSON Schema; a manual change is silently overwritten the
next time anyone regenerates them, and a models-only diff is rejected in
review (INV-003). This page explains what they are and how to change
them correctly.

## What this directory is

These modules are the in-memory Python form of a HilbertBench trace: the
manifest, spans, artifact metadata, and the catalog. They are compiled
from the JSON schemas in [`../../schemas/v1.0/`](../../schemas/v1.0/) by
`datamodel-code-generator`. The schema is the source of truth and the
Python is build output — the same relationship an object file has with
its C source.

```
schemas/v1.0/*.json   ──(generate_models.py)──►   hilbertbench/models/v1_0/*.py
   the source of truth                               generated, do-not-edit
```

Every generated file opens with a banner that says so:

```python
# AUTO-GENERATED BY datamodel-codegen — DO NOT EDIT MANUALLY
# Source: schemas/v1.0/
# Regenerate: python schemas/scripts/generate_models.py
# INV-003: Manual edits will be overwritten. Change the schema, not this file.
```

## Why it works this way

A trace must be readable by tools written in **any** language, years
from now, long after this Python package may have changed. If the data
model lived only in Python, the schema would be whatever the code
happened to do — undocumented and unportable. By making a
language-agnostic JSON Schema the source of truth:

- the trace format is **specified independently** of any
  implementation (INV-003);
- a tagged schema version is **frozen forever** so old traces always
  parse (INV-005);
- a Rust, Julia, or JavaScript reader can be generated from the same
  schema, enabling the "publish your traces, analyse them anywhere"
  model the project is built around.

Hand-editing the Python would fork the model away from the
specification without anyone noticing — the precise failure this design
rules out.

## How to change a model

1. Edit the JSON schema, not the Python. Pick the file in
   [`../../schemas/v1.0/`](../../schemas/v1.0/) that owns the field
   (`trace.json`, `span.json`, `artifact.json`, or `catalog.json`); the
   [schema guide](../../schemas/README.md) covers them field by field.
2. Watch the version freeze (INV-005). If `schemas/v1.0/` has already
   been tagged (`schema-v1.0`) you cannot touch it in place — create
   `schemas/v1.1/` for the new shape. Editing `v1.0/` directly is only
   allowed while it is still unreleased.
3. Regenerate:
   ```bash
   python schemas/scripts/generate_models.py
   ```
   The script rewrites every file in `v1_0/`, restores the banner, and
   runs enum and field sanity checks.
4. Run the boundary tests, which catch a model/schema mismatch at once:
   ```bash
   python -m pytest tests/trace tests/recorder -q
   ```
5. Commit the schema change and the regenerated models in the same
   commit. A models-only diff with no accompanying schema change is the
   signature of a hand-edit and gets flagged.

## What you can safely edit here

Only [`__init__.py`](__init__.py), which is not generated. It is the
version-stable export surface (`from hilbertbench.models import
HilbertbenchSpan, Kind, ...`) that lets callers avoid importing `v1_0`
directly, so a future `v1_1` can be swapped in underneath them.
Everything inside `v1_0/` is generated.

## Layout

| Path | Source schema | Contents |
|------|---------------|----------|
| `v1_0/trace.py` | `trace.json` | trace manifest, mode, status, integrity seal, client environment |
| `v1_0/span.py` | `span.json` | execution spans, events, inline artifacts |
| `v1_0/artifact.py` | `artifact.json` | artifact metadata, `Kind`/`Encoding`/`Compression` enums |
| `v1_0/catalog.py` | `catalog.json` | the artifact catalog index |
| `__init__.py` | — | hand-written public re-exports (safe to edit) |
